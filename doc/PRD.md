# Expedia 酒店表格生成工具 - 产品需求文档 (PRD)

## 一、项目概述

### 1.1 项目背景
亚朵渠道运营团队需要将酒店信息上架至 Expedia 平台，目前需要手动整理大量数据、翻译内容、填充表格，效率低下且容易出错。本工具旨在自动化这一流程，提升运营效率。

### 1.2 目标用户
- 亚朵渠道运营伙伴
- 酒店数据维护人员
- 翻译审核人员

### 1.3 核心价值
- **效率提升**：自动化数据组合与填充，减少 80% 手动操作
- **翻译质量**：AI 多源参考润色，确保翻译专业准确
- **合规保障**：遵循 Expedia 平台规范，减少上架驳回率

### 1.4 项目范围
- 支持酒店主数据和客房主数据的导入与管理
- 提供智能翻译功能，支持多源参考和规则约束
- 自动生成符合 Expedia 规范的上传表格
- 支持翻译审核和术语库管理

### 2.2 翻译模块

#### 2.2.1 翻译规则管理
翻译规则可以由多项规则构成：公司规则、行业规则、国家颁布的规则等。

**规则层级**:
- 用户可以上传多个规则，并为这些规则添加适用的省、市、国家
- 翻译不同省市酒店时优先参考公司规则，其次参考国家、省、市的各项规则

**规则来源**:
- 集团翻译规则: `reference/SOP-酒店房型英文翻译规则1.0.pdf`
- 其他翻译规则以 PDF 格式呈现
- 上传 PDF 后系统自动解析，通过大模型理解含义并生成翻译规则

#### 2.2.2 翻译参考库
翻译时优先参考携程和 Booking 的翻译，符合要求则直接使用，否则使用翻译规则进行翻译。

**参考数据**:
- Booking/携程酒店主数据翻译: `reference/Booking亚朵集团酒店主数据英文信息--3.25.xlsx`
- Booking 房间主数据翻译: `reference/Booking亚朵集团房型英文信息核查-3.25.xlsx`

#### 2.2.3 翻译工作流
```
翻译处理流程:
1. 查询携程参考库
   ├── 查到 → 通过公司规范、各级行政规范进行大模型校验并润色
   └── 未命中 → 继续下一步
   
2. 检测源文本语言，调用腾讯云翻译 API
   ├── 翻译成功 → 通过规范进行大模型校验并润色
   └── 翻译不可用 → 继续下一步
   
3. 直接通过大模型结合规范进行翻译
```

#### 2.2.4 翻译质量评估

| 评估维度 | 说明 | 指标 |
|---------|------|------|
| 准确性 | 语义是否准确传达 | 人工抽检 |
| 专业性 | 术语使用是否规范 | 术语库匹配率 |
| 本土化 | 表达是否符合目标市场习惯 | AI 评分 |
| 完整性 | 信息是否完整无遗漏 | 字段覆盖率 |

#### 2.2.5 人工审核功能
| 功能项 | 说明 |
|-------|------|
| 翻译对比 | 原文/译文/参考来源对照展示 |
| 批量审核 | 支持批量接受/拒绝/修改 |
| 历史记录 | 翻译修改历史追溯 |
| 术语库更新 | 审核确认后自动更新术语库 |

### 2.3 Expedia 数据配置模块

#### 2.3.1 数据模型
根据 Expedia 官方提供的模板生成数据模型。

**参考文件**:
- Expedia 上线表格: `reference/Expedia 亚朵集团酒店上线表格 - APM Onboarding Sheet V5 EN & CN Batch4-亚朵灰度门店1家.xlsx`
- 酒店邮箱信息: `reference/【飞书正式环境】公共邮箱_数据表 (2).xlsx`

#### 2.3.2 数据合并引擎
```
处理流程:
1. 加载酒店主数据
2. 加载客房主数据
3. 通过 hotel_id 建立关联
4. 执行字段映射规则
5. 生成组合数据集
6. 标记缺失字段
```

#### 2.3.3 缺失数据识别与填充
| 缺失类型 | 处理策略 |
|---------|---------|
| 必填字段缺失 | 标红警告，阻止导出 |
| 推荐字段缺失 | 黄色提示，可继续导出 |
| 可选字段缺失 | 灰色标记，可忽略 |

**自动填充规则**:
- Country: 默认 "China"
- Currency: 默认 "CNY"
- Check-in Time: 默认 "14:00"
- Check-out Time: 默认 "12:00"
- Property Type: 根据星级判断（≥4星→Hotel，<4星→Lodging）
- Language: 默认 "zh-CN" + "en-US"

## 五、数据校验规则

### 5.1 酒店数据校验

#### 5.1.1 必填字段
- hotel_id（业务ID）
- name_zh（中文名称）
- address_zh（中文地址）
- city（城市）
- province（省份）
- phone（联系电话）
- email（邮箱）
- latitude（纬度）
- longitude（经度）

#### 5.1.2 格式校验规则
| 字段 | 校验规则 | 错误提示 |
|-----|---------|---------|
| hotel_id | 唯一性校验，不能重复 | 酒店ID已存在 |
| phone | 手机号或座机格式 | 电话格式不正确 |
| email | 标准邮箱格式 | 邮箱格式不正确 |
| latitude | 范围: -90 到 90 | 纬度超出范围 |
| longitude | 范围: -180 到 180 | 经度超出范围 |
| star_rating | 范围: 1-5 或空 | 星级必须是1-5 |

#### 5.1.3 业务逻辑校验
- 省市匹配校验
- 邮箱域名白名单（atour.com, atahouse.com）

### 5.2 客房数据校验

#### 5.2.1 必填字段
- hotel_id（关联酒店ID）
- room_type_zh（房型中文名）
- bed_type（床型）
- bed_count（床数量）
- max_occupancy（最大入住人数）

#### 5.2.2 关联校验
| 校验项 | 规则 | 错误提示 |
|-------|------|---------|
| hotel_id 存在性 | 必须在 hotels 表中存在 | 关联的酒店不存在 |
| hotel_id 状态 | 关联酒店状态必须为 active | 关联的酒店未激活 |
| 房型唯一性 | 同一酒店下房型名称不能重复 | 该酒店已存在相同房型 |

#### 5.2.3 数值范围校验
| 字段 | 范围 | 说明 |
|-----|------|------|
| area | 10-500 | 面积(平方米) |
| bed_count | 1-10 | 床数量 |
| max_occupancy | 1-20 | 最大入住人数 |

### 5.3 翻译数据校验

- 译文长度不应小于原文的 30%
- 译文长度不应超过原文的 3 倍
- 英文翻译不应包含中文字符

## 六、非功能性需求

### 6.1 性能要求
| 指标 | 要求 |
|-----|------|
| 页面加载时间 | < 3 秒 |
| 数据导入速度 | > 1000 条/分钟 |
| 翻译处理速度 | > 100 条/分钟 |
| 并发用户数 | 支持 50 人同时在线 |

### 6.2 可用性要求
- 系统可用性: 99.5%
- 数据备份: 每日备份
- 故障恢复时间: < 1 小时

### 6.3 兼容性要求
- 浏览器: Chrome 90+、Edge 90+、Safari 14+
- 分辨率: 最小支持 1280x720
